# VatBrain — Roadmap

> 从"被动知识库"到"主动记忆系统"的演进路线。
> 每个版本有明确的交付物、依赖关系和准入/准出标准。

---

## 版本总览

```
v0.1 ──── v0.2 ──── v0.3 ──── v1.0 ──── v1.x
最小闭环   记忆进化    预测能力    多智能体   持续演进
(2-3周)   (2-3周)    (3-4周)    (4-6周)    (长期)
```

| 版本 | 主题 | 核心交付 |
|-----|------|---------|
| v0.1 | 最小闭环 | 图+向量复合写入、两阶段检索、权重衰减、睡眠整合 v1 |
| v0.2 | 记忆进化 | 错误记忆独立建模、记忆再巩固、行为归因权重 |
| v0.3 | 预测与主动 | 风险预测、反事实推理、预测误差信号 |
| v1.0 | 多智能体记忆 | 跨 Agent 记忆共享、记忆冲突协调、多租户 |
| v1.x | 持续演进 | 联邦记忆、自适应衰减参数、长期评估体系 |

---

## v0.1 — 最小闭环

**目标**：跑通"写入 → 衰减 → 检索 → 整合"的完整链路。验证架构可行性。

### 交付物

| 模块 | 内容 | 完成标准 |
|-----|------|---------|
| 基础设施 | Neo4j + pgvector + Redis + MinIO docker-compose | 一键启动，健康检查通过 |
| 数据模型 | EpisodicMemory / SemanticMemory 节点与边 | Schema 可执行 Cypher 创建 |
| 写入链路 | 显著性门控 → 可分离性判别 → 复合写入 | 单条写入 < 200ms |
| 权重引擎 | Recency-Weighted Frequency + 双参照衰减 + 冷却阈值 | 衰减曲线与预期误差 < 5% |
| 检索链路 | 情境过滤（硬约束+软权重）→ 语义排序 | 检索延迟 < 100ms(命中)/< 500ms(miss) |
| 睡眠整合 v1 | 定时触发，回顾→提炼→回测→持久化 | 单次整合 < 5min (1000条) |
| API 层 | 写入/检索/反馈/整合触发 | 所有端点有 OpenAPI 文档 |
| MCP 集成 | MCP Server 封装，对接到至少 1 个 AI Agent | 端到端可用 |

### 不入

- 错误记忆独立节点
- 记忆再巩固（检索反向传播）
- 反事实推理
- 预测误差信号
- 多租户

### 准入标准
- 开发环境就绪
- Neo4j + pgvector 实例可用
- LLM API key 配置完成

### 准出标准
- 全部单元测试通过（覆盖率 > 85%）
- 集成测试：写入 100 条 → 检索 → 衰减 → 整合端到端通过
- 性能基准达标
- 至少 1 个 AI Agent 可正常调用 MCP 工具

### 关键风险
- 睡眠整合中 LLM 提炼规则的质量不稳定 → 降低回测阈值至 0.6，人工 review 首批规则
- pgvector IVFFlat 在大数据量下性能下降 → v0.1 数据量可控，v0.3 考虑迁移 Milvus

## v0.2 — 记忆进化

**目标**：让记忆系统从"被动记录"进化为"从错误中学习"。错误记忆独立建模是这一版的核心。

### 前置依赖
- v0.1 准出通过，积累至少 2 周的运行数据
- 情境记忆库中有足够 `task_type=debug` 的记录（建议 > 200 条）

### 交付物

| 模块 | 内容 | 完成标准 |
|-----|------|---------|
| Pitfall 节点模型 | 独立 Neo4j 节点类型，包含 signature / root_cause_category / fix_strategy / was_user_corrected | Schema 定义完成，可查询 |
| Pitfall 提取引擎 | 睡眠整合中新增 Pitfall 提取：从 debug 类型情境记忆聚类 → LLM 结构化 → 写入 Pitfall 节点 | 召回率 > 70%（人工标注验证集） |
| 记忆再巩固 | 检索命中 → 用户纠正 → 反向传播更新源记忆权重和内容 | 纠正链溯源可验证 |
| 行为归因权重 | 替换 v0.1 的静态评分：权重增量由后续行为（引用/纠正/确认）驱动 | 权重更新延迟 < 1min |
| 错误感知检索 | 检索时，对目标实体的 Pitfall 记忆做双键匹配（实体 + 情境签名） | 错误记忆命中率提升可量化 |
| 源监控增强 | EpisodicMemory 和 SemanticMemory 的 `source_type` / `trust_level` 加入检索排序 | 高可信源记忆排序优先 |

### 不入

- 风险预测（Pitfall → 主动预警）
- 反事实推理
- 多租户

### 准出标准
- Pitfall 提取：在 200 条 debug 情境记忆中至少提取出 10 条有效 Pitfall 规则
- 记忆再巩固：纠正 10 条错误检索结果，源记忆权重正确更新
- 行为归因：权重增量与人工预期偏差 < 15%
- 检索质量对比：v0.2 vs v0.1 在调试场景下的首条命中率有提升

### 关键风险
- 数据积累不足（debug 记录不够）→ 人工注入一批历史调试记录做种子数据
- Pitfall 提取噪声大 → 提高聚类阈值，初始 MinClusterSize=3，若噪声大调高至 5（详见 v0.2 D2 决策）

## v0.3 — 预测与主动

**目标**：记忆系统从"被查时回答"升级为"在需要前主动提醒"。预测能力是这个版本的分水岭。

### 前置依赖
- v0.2 准出通过
- Pitfall Memory 积累 > 50 条有效规则
- 至少 1 个活跃项目持续使用 >= 1 个月

### 交付物

| 模块 | 内容 | 完成标准 |
|-----|------|---------|
| 风险评分引擎 | 对即将修改的模块计算风险得分（历史 Pitfall 密度 × 修复时间衰减 × 模块复杂度） | 高风险模块识别准确率 > 60% |
| 主动注入 | Agent 进入高危模块时，自动将相关 Pitfall 注入 Working Memory（不等 Agent 检索） | 注入延迟 < 200ms |
| 预测误差信号 | 独立 Surprise Score 维度：预期被打破的事件获得 3x 半衰期 | 高惊讶度事件 7 天后权重显著高于同期低惊讶度事件 |
| 反事实推理 | 睡眠阶段新增：用户连续 N 次纠正同一模式 → 生成假设性规则（不等用户确认，标记为 proposed） | 假设性规则对用户可见，可一键确认/拒绝 |
| 压缩残差存储 | Level 2→3 压缩时存储 embedding 残差向量；检索时摘要不够精确可"解压"回 Level 2 | 解压命中率 > 80% |
| 自适应衰减参数 | α/β 参数基于项目特征自动微调（活跃项目衰减更慢，死项目加速冷却） | 参数调整有迹可循，人工可覆盖 |

### 不入

- 多智能体
- 联邦记忆

### 准出标准
- 主动注入：在 10 次"进入高危模块"场景中，>= 6 次注入的 Pitfall 被 Agent 实际使用
- 反事实规则：用户确认率 > 40%（> 40% 的 proposed 规则被确认有效）
- 预测误差：高惊讶度事件的 7 日存留率显著高于对照组

### 关键风险
- 主动注入过多导致干扰 → 严格限制注入条数（最多 3 条），降权显示
- 反事实推理产生大量垃圾规则 → proposed 规则 14 天无确认自动清除

## v1.0 — 多智能体记忆

**目标**：多个 Agent（不同开发者、不同 AI 助手）共享记忆，但各自维护独立的情境视图。

### 前置依赖
- v0.3 准出通过
- 至少 3 个以上活跃用户/Agent 使用

### 交付物

| 模块 | 内容 |
|-----|------|
| 多租户架构 | project → team → agent 三级隔离；硬约束增加 `team_id` |
| 记忆共享协议 | 语义记忆全局共享（同一项目），情境记忆 Agent 私有（默认） |
| 冲突协调引擎 | 不同 Agent 贡献的语义规则冲突时：信任度高的覆盖低的，同等信任的人工裁决 |
| 跨 Agent 学习 | Agent A 的 Pitfall → Agent B 在同一实体上可被动受益 |
| 记忆订阅 | Agent 可订阅某个模块的记忆更新，该模块有新 Pitfall/规则时推送 |
| 联合检索 | 检索时可选范围：仅自己的 / 团队的 / 全局的 |

### 准出标准
- 3 个 Agent 并行使用，情境记忆不互相污染
- 语义记忆共享：A 贡献的规则，B 可检索到
- 冲突协调：同时存在矛盾规则时，按 trust_level 正确排序

## v1.x — 持续演进

不设硬性时间节点，按需推进。

| 方向 | 描述 |
|-----|------|
| 联邦记忆 | 跨团队/跨组织的记忆安全共享（脱敏 + 差分隐私） |
| 自适应衰减 | 衰减参数完全由数据驱动，每个 entity_group 有独立衰减曲线 |
| 记忆健康评估 | 定期评估记忆库的"信噪比"、"覆盖率"、"时效性"，给出清理建议 |
| 长期评估体系 | 对比使用 VatBrain 和未使用 VatBrain 的 Agent 在任务完成时间、错误率上的差异 |
| 记忆可视化 | D3/Graphviz 可视化记忆图谱、衰减曲线、关联路径 |
| 知识蒸馏 | 从大量情境记忆蒸馏出"最佳实践"级别的语义规则，直接可嵌入系统提示 |

## 跨版本关注事项

### 性能演进

| 指标 | v0.1 目标 | v0.3 目标 | v1.0 目标 |
|-----|----------|----------|----------|
| 情境记忆总量 | < 10K | < 100K | < 1M |
| 写入延迟 | < 200ms | < 150ms | < 100ms |
| 检索延迟（缓存命中） | < 100ms | < 80ms | < 50ms |
| 检索延迟（缓存 miss） | < 500ms | < 300ms | < 200ms |
| 睡眠整合耗时 | < 5min | < 15min | < 30min（增量） |

### 数据留存策略

| 记忆类型 | v0.1 | v0.2+ | 冷却后处理 |
|---------|------|-------|-----------|
| 热记忆 | Redis | Redis | → pgvector 冷分区 |
| 温记忆 | pgvector | pgvector | 正常衰减 |
| 冷记忆 | pgvector 冷分区 | pgvector 冷分区 | 7 天后归档 MinIO |
| 归档 | MinIO (Level 1) | MinIO (Level 1) | 90 天后可配置删除 |

### 技术债备忘录

- pgvector 到 Milvus 的迁移点：单表 > 100K 向量或检索延迟 > 500ms 时启动
- Neo4j 分片：单实例 > 1M 节点时考虑
- 睡眠整合从"定时"到"闲置检测"的切换点：积累足够的 Agent 活跃度数据后（v1.0+）
